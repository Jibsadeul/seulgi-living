# member-info 기능 명세

## 목적

`member-info` feature는 사용자 기본 정보 바텀시트에서 필요한 입력, 검증, 닉네임 중복 확인, 지역 선택, 저장 플로우를 담당한다.

이 feature는 화면 진입 여부를 결정하지 않는다. 바텀시트가 언제 열리는지는 `screens/members`에서 결정한다.

## 범위

### features/member-info

- 닉네임 입력 상태를 관리한다.
- 닉네임 중복 확인 요청과 결과 상태를 관리한다.
- 생년월일 년/월/일 선택 상태를 관리한다.
- 시/도와 시/군/구 선택 상태를 관리한다.
- 시/도 변경 시 시/군/구 선택값을 초기화한다.
- 전체 폼 유효성, 저장 가능 여부, 닫기 가능 여부를 계산한다.
- 사용자 기본 정보 저장 mutation을 호출한다.

## 입력 모델

| 필드       | 타입   | 설명          |
| ---------- | ------ | ------------- |
| nickname   | string | 사용자 닉네임 |
| birthYear  | string | 생년          |
| birthMonth | string | 생월          |
| birthDay   | string | 생일          |
| sidoId     | string | 시/도 ID      |
| sigunguId  | string | 시/군/구 ID   |

저장 요청 전 `birthYear`, `birthMonth`, `birthDay`를 조합해 `YYYY-MM-DD` 형식의 `birthday` 값을 만든다.

## 초기값

feature는 현재 사용자 기본 정보를 초기값으로 받을 수 있어야 한다.

| 입력      | 초기값 처리                                  |
| --------- | -------------------------------------------- |
| nickname  | 기존 닉네임이 있으면 입력값으로 설정한다.    |
| birthday  | 기존 생년월일이 있으면 년/월/일로 분리한다.  |
| sidoId    | 기존 거주지의 시/도 ID가 있으면 선택한다.    |
| sigunguId | 기존 거주지의 시/군/구 ID가 있으면 선택한다. |

기존 닉네임이 있고 변경되지 않은 경우 닉네임 중복 확인 상태는 사용 가능으로 간주할 수 있다.

## 닉네임 검증

- 닉네임은 앞뒤 공백을 제거한 값을 기준으로 검증한다.
- 닉네임은 2자 이상이어야 한다.
- 닉네임은 DB `varchar(100)` 기준에 맞춰 100자 미만이어야 한다.
- 닉네임이 비어 있거나 길이 조건을 만족하지 않으면 중복 확인 버튼을 비활성화한다.
- 닉네임이 변경되면 기존 중복 확인 성공 상태를 무효화한다.
- 중복 확인 성공 후 입력값이 다시 바뀌면 저장할 수 없다.

## 생년월일 검증

- 년, 월, 일이 모두 선택되어야 한다.
- 조합된 값은 실제 존재하는 날짜여야 한다.
- 예: 2월 30일은 유효하지 않다.
- 저장 요청에는 `YYYY-MM-DD` 문자열을 전달한다.
- 서버와 DB는 이 값을 PostgreSQL `date`로 저장 가능한 값으로 검증한다.

## 지역 선택

- 최초 렌더링 시 시/도 목록을 조회한다.
- 시/도를 선택하면 해당 `sidoId`에 속한 시/군/구 목록을 조회한다.
- 시/도 선택 전에는 시/군/구 셀렉트 박스를 비활성화한다.
- 시/도를 변경하면 기존 `sigunguId`를 빈 값으로 초기화한다.
- 시/군/구 목록 조회 실패 시 시/군/구 셀렉트 박스에 오류 상태를 표시하고 저장을 막는다.

## 저장 가능 조건

다음 조건을 모두 만족해야 저장 가능하다.

- 닉네임 길이 검증 통과
- 닉네임 중복 확인 통과
- 유효한 생년월일
- 시/도 선택 완료
- 시/군/구 선택 완료
- 지역 목록 조회 오류 없음
- 저장 요청 진행 중 아님

## 닫기 가능 조건

- onboarding 모드에서는 feature가 닫기 가능 상태를 반환하더라도 screen에서 저장 성공 전 닫기를 막는다.
- edit 모드에서는 폼이 유효한 경우에만 닫기 가능 상태로 본다.

## 저장 플로우

1. 저장 버튼을 누른다.
2. feature가 현재 폼 상태를 다시 검증한다.
3. `birthday`를 `YYYY-MM-DD`로 조합한다.
4. 사용자 기본 정보 저장 mutation을 호출한다.
5. 성공 시 갱신된 사용자 정보를 상위 화면에 전달한다.
6. 실패 시 오류 메시지를 노출하고 폼 상태를 유지한다.

## 오류 메시지 기준

| 조건                   | 메시지 의미                     |
| ---------------------- | ------------------------------- |
| 닉네임 2자 미만        | 닉네임은 최소 2자 이상이어야 함 |
| 닉네임 100자 이상      | 닉네임은 100자 미만이어야 함    |
| 닉네임 중복            | 이미 사용 중인 닉네임           |
| 생년월일 미완성        | 년, 월, 일을 모두 선택해야 함   |
| 생년월일 유효하지 않음 | 존재하지 않는 날짜              |
| 시/도 미선택           | 시/도를 선택해야 함             |
| 시/군/구 미선택        | 시/군/구를 선택해야 함          |
| 저장 실패              | 서버 저장 실패 안내             |

## 검증 기준

- 닉네임 변경 시 중복 확인 상태가 초기화된다.
- 닉네임 중복 확인 전에는 저장할 수 없다.
- 시/도 변경 시 시/군/구 값이 초기화된다.
- 유효하지 않은 날짜는 저장할 수 없다.
- 모든 입력값이 유효하고 닉네임 중복 확인이 완료된 경우에만 저장할 수 있다.
